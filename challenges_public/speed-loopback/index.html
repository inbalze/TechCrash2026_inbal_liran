Speed Loopback

Transfer 10,000 bytes from the FPGA to the ESP32 and back. The fastest team wins.

The Challenge

The FPGA generates 10,000 pseudo-random bytes and sends them to the ESP32. The ESP32 must receive all 10,000 bytes, compute the sum (keeping only the lowest 8 bits), and send that single checksum byte back to the FPGA.

The FPGA independently computes the expected checksum. When it receives the ESP32's answer, it stops the timer and compares. If the checksum matches: PASS. The 7-segment displays show the elapsed time in milliseconds (hex).

Scoring: Lowest elapsed time (in ms) with a correct checksum wins. Incorrect checksums score zero. The baseline implementation runs at 9600 baud and takes roughly 10.4 seconds. Your job: obliterate that number.

FPGA Architecture

The FPGA contains fixed infrastructure (red) that you must not modify, and a communication layer (green) that is yours to replace.

DO NOT MODIFY — Fixed infrastructure
YOUR CODE — Communication layer (replace/upgrade)

Rules

DO NOT MODIFY the fixed FPGA infrastructure: LFSR random generator, sum accumulator, millisecond timer, comparator, state machine, or the data count (10,000). Changing any of these disqualifies your run.
YOU MAY MODIFY anything in the communication path:
  • Replace the UART TX/RX with faster UARTs, SPI, parallel bus, or anything else
  • Change baud rates
  • Add multiple communication channels in parallel
  • Rewrite the entire ESP32 firmware
  • Use any GPIO pins on the JP1 header

Protocol

  1. FPGA sends a 4-byte header: the count N as a 32-bit little-endian integer (always 10,000 = 0x00002710)
  2. FPGA sends N data bytes: pseudo-random values from a 16-bit LFSR
  3. ESP32 computes sum = (byte0 + byte1 + ... + byteN-1) & 0xFF
  4. ESP32 sends back 1 byte: the checksum
  5. FPGA stops the timer and compares

How to Run

  1. Flash the FPGA with the provided speed_loopback_top.sof
  2. Flash the ESP32 with your firmware
  3. Connect the wires between JP1 and ESP32
  4. Press KEY[0] to start a run
  5. Wait for LEDR[9] to turn off (run complete)
  6. Read the result:
    • LEDR[0] = green = PASS (checksum correct)
    • LEDR[1] = red = FAIL (checksum mismatch)
    • HEX5..0 = elapsed time in ms (hexadecimal)
  7. Toggle SW[9] up to see debug info: expected checksum (HEX3..2) vs received (HEX1..0)
  8. Press KEY[0] again to re-run

Optimization Ideas

  • Increase baud rate: Change the UART baud parameter in both FPGA and ESP32. The FPGA can do any baud. The ESP32 supports up to ~5 Mbaud.
  • Multiple UARTs: The ESP32 has UART1 and UART2 available. Instantiate two UART TX modules on the FPGA, each sending half the data on different GPIO pins.
  • SPI: Use the ESP32 as an SPI slave. SPI can clock at 10+ MHz with full-duplex capability. You'll need to write an SPI master on the FPGA.
  • Parallel bus: Use 8 GPIO pins as a parallel data bus with a clock and handshake signal. Maximum throughput but uses the most wires.
  • Minimize ESP32 overhead: Disable the OLED updates during data transfer. Use DMA if available. Process data in bulk, not byte-by-byte.
Hint: The biggest wins come from changing the physical interface (UART → SPI/parallel), not from micro-optimizing the ESP32 code. Focus on the communication layer first.
