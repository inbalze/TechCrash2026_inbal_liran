FP8 Adder Race

You get a correct but intentionally slow FP8 E4M3 adder. The board runs 4096 locked test vectors, lets you tune the DUT/test clock with a PLL, and measures the final runtime on a separate fixed 25 MHz stopwatch path. Your job is to make the result faster without breaking correctness.

Overview

You are given a working starter design for an FP8 E4M3 adder challenge. The reference DUT is intentionally slow and non-pipelined. The rest of the system is already built for you: ROMs with operand pairs, a controller that launches every test, a checker that compares against golden results, and a board UI that reports the final runtime.

The competition now has two knobs: your adder micro-architecture and the DUT/test clock generated by an editable PLL. The stopwatch is still locked. A separate fixed 25 MHz measurement clock measures real elapsed time in microseconds, so teams can push frequency without touching the timing reference.

FP8 E4M3 Format

Bit layout:  [7]  [6:3]   [2:0]
             sign  exp     mantissa

Bias = 7
Normal:   (-1)^s  x  2^(exp-7)  x  1.mantissa
Denormal: (-1)^s  x  2^(1-7)    x  0.mantissa   (exp = 0000)
NaN:      exp = 1111, mantissa = 111  (0x7F or 0xFF)
No infinity in E4M3 (exp=1111 with mantissa != 111 is a valid number)

Max value:  +/- 448  (0x7E / 0xFE)
Min normal: +/- 2^-6 = 0.015625
Rounding:   Round-to-nearest-even (IEEE default)

Rules

  • You may modify fp8_adder.v and challenge_pll.v
  • You may add new modules (instantiated from fp8_adder.v)
  • You may change the architecture to pipelined, combinational, or any other valid implementation
  • You may tune the DUT/test clock frequency through the editable PLL
  • You must keep the same port interface (clk, rst_n, start, a, b, result, done, busy)
  • Do NOT modify fp8_top.v, test_controller.v, rom_8x4096.v, or seg7_hex.v
  • Do NOT modify the fixed 25 MHz measurement clock path, board pins, or test vectors
  • Do NOT modify the .hex test vector files
  • Do NOT hardcode results or bypass computation

Safe PLL Tuning

The PLL is editable on purpose, but treat it like a controlled experiment. Push it in small steps, keep the stopwatch path untouched, and let Quartus plus the board tell you when you crossed the line.

// challenge_pll.v
localparam integer CLK0_MULTIPLY_BY = 1;
localparam integer CLK0_DIVIDE_BY   = 2;

// DUT/test clock = 50 MHz * CLK0_MULTIPLY_BY / CLK0_DIVIDE_BY

Adder Interface

module fp8_adder (
    input  wire       clk,      // Editable DUT/test clock from challenge_pll.v
    input  wire       rst_n,    // Active-low reset
    input  wire       start,    // Pulse high 1 cycle to begin
    input  wire [7:0] a,        // Operand A (stable while busy)
    input  wire [7:0] b,        // Operand B (stable while busy)
    output reg  [7:0] result,   // FP8 sum (valid when done=1)
    output reg        done,     // Pulses high 1 cycle when complete
    output reg        busy      // High while computing
);

Protocol

  • Test controller asserts start for 1 clock with valid a, b
  • Adder raises busy while the operation is in flight
  • Adder pulses done for 1 clock with valid result
  • The controller does not launch the next test until your adder finishes the current one
  • NaN comparison: any NaN output matches any NaN expected

Board Display

While running: HEX5 HEX4 = E E LEDR[9:1] = progress bar LEDR[0] = 0 When finished: HEX5..HEX0 = total elapsed time in decimal microseconds LEDR[0] = 1 if all 4096 tests passed, 0 otherwise LEDR[9:1] = 0 Controls: KEY[0] = Start (active low, press to begin test) KEY[1] = Reset (active low, press to reset) SW[9:0] are unused by the locked harness

Starter Code Map

The starter is intentionally split into two zones: a small editable DUT and a larger locked harness. If you stay inside that boundary, your work is valid. If you start changing the harness, you are changing the benchmark instead of solving it.

Editable Core

fp8_adder.v is the reference implementation. It is correct, stable, and slow on purpose. Replace the state machine, shorten the latency, pipeline it, or fully redesign it. challenge_pll.v lets you tune the DUT/test clock as well.

Locked Harness

fp8_top.v owns the board UI and the fixed measurement path. test_controller.v owns the handshake and golden checks. The ROM files and wrappers define the official test set.

Getting Started

  1. Open challenges/fp8-adder/fpga/ in Quartus Prime Lite 17.1
  2. Run the provided simulation and make sure the starter passes all 4096 vectors before you touch anything
  3. Compile and program the board, press KEY[0], and record the starter runtime shown on HEX5..HEX0
  4. Open src/fp8_adder.v and src/challenge_pll.v to optimize both the architecture and the DUT/test clock
  5. After each change, rerun simulation first, then recompile and check the board runtime
  6. Your first milestone is simple: keep LEDR[0] = 1 and cut the runtime to half of the starter
  7. After that, keep iterating for the fastest correct board result you can get

Hints

What Actually Matters

Now both latency and frequency matter. The stopwatch is fixed, but the DUT/test path is not, so the winning design is the one that turns real elapsed time into the smallest number on the board.

Ideas

  • The controller waits for done before starting the next test, so per-operation latency matters directly.
  • Higher DUT frequency helps only if your logic still meets timing. Treat the PLL and the architecture as one system.
  • If you pipeline, make sure the handshake still behaves exactly as the harness expects.
  • Keep the Python model nearby when you touch rounding, denormals, NaNs, signed zero, or overflow saturation.
  • Use the simulation testbench aggressively. It is much cheaper than finding out on the board that you broke one corner case.
  • Do not waste time editing the fixed measurement path. It is locked, and it is already measuring the thing the judges care about.
